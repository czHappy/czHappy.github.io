CUDA 简介

  • CUDA(Compute Unified Device Architecture，统一计算设备架构)是由 NVIDIA 推出的通用并行计算架构，该架构使 GPU 能够解决复杂的计算问题。 它包含了 CUDA 指令集架构（ISA）以及 GPU 内部的并行计算引擎。。CUDA不仅是一个硬件技术，还包含一套完整的软件生态系统，为开发者提供了从底层编程接口到高层应用框架的一系列工具，使得非图形领域的应用程序能够利用GPU进行加速。

计算机架构

冯诺伊曼体系

  • 三大部分
    • 内存（指令内存，数据内存）
    • 中央处理单元（控制单元和算数逻辑单元）
    • 输入、输出接口

术语

  • SISD: 单指令单数据SISD（传统串行计算机，386）
  • SIMD: 单指令多数据SIMD（并行架构，比如向量机，所有核心指令唯一，但是数据不同，现在CPU基本都有这类的向量指令
  • MISD: 多指令单数据MISD（少见，多个指令围殴一个数据）
  • MIMD:多指令多数据MIMD（并行架构，多核心，多指令，异步处理多个数据流，从而实现空间上的并行，MIMD多数情况下包含SIMD，就是MIMD有很多计算核，计算核支持SIMD）

指标/瓶颈

  • 降低延迟
    • 延迟是指操作从开始到结束所需要的时间，一般用微秒计算，延迟越低越好
  • 提高带宽
    • 带宽是单位时间内处理的数据量，一般用MB/s或者GB/s表示。
  • 提高吞吐量
    • 吞吐量是单位时间内成功处理的运算数量，一般用gflops来表示（十亿次浮点计算）
    • 吞吐量和延迟有一定关系，都是反应计算速度的
      • 一个是时间除以运算次数，得到的是单位次数用的时间–延迟，
      • 一个是运算次数除以时间，得到的是单位时间执行次数–吞吐量。

异构计算

CPU与GPU的区别

  • GPU包括更多的运算核心，其特别适合数据并行的计算密集型任务，如大型矩阵运算，而CPU的运算核心较少，但是其可以实现复杂的逻辑运算，因此其适合控制密集型任务。另外，CPU上的线程是重量级的，上下文切换开销大，但是GPU由于存在很多核心，其线程是轻量级的。因此，基于CPU+GPU的异构计算平台可以优势互补，CPU负责处理逻辑复杂的串行程序，而GPU重点处理数据密集型的并行计算程序，从而发挥最大功效。GPU无论发展得多快，都只能是替CPU分担工作，而不是取代CPU。

异构架构

  • 定义：x86 CPU+GPU是最常见的异构架构，也有CPU+FPGA，CPU+DSP等各种各样的组合

  • 架构异构

  • 代码异构

    • 主机代码在主机端运行，被编译成主机架构的机器码
    • 设备端的在设备上执行，被编译成设备架构的机器码，所以主机端的机器码和设备端的机器码是隔离的，自己执行自己的，没办法交换执行。
    • 主机端代码主要是控制设备，完成数据传输等控制类工作，设备端主要的任务就是计算。

线程

  • CPU线程是重量级实体，操作系统交替执行线程，线程上下文切换花销很大
  • GPU线程是轻量级的，GPU应用一般包含成千上万的线程，多数在排队状态，线程之间切换基本没有开销。
  • CPU的核被设计用来尽可能减少一个或两个线程运行时间的延迟，而GPU核则是大量线程，最大幅度提高吞吐量

    • 设备：GPU及其内存

线程管理

  • 概括
    • 一个核函数只能有一个grid
    • 一个grid可以有很多个块
    • 每个块可以有很多的线程

  • 线程块block

    • 协作

      • 同步
      • 共享内存

    • 不同块内线程不能相互影响，他们是物理隔离的

  • 线程编号

    • 两个内置结构体确定线程标号
      • blockIdx，可以认为是网格内的块偏移量（线程块在线程网格内的位置索引，有x,y,z三个分量，代表三个方向的偏移量，基于 uint3 定义，包含三个无符号整数的结构）
      • threadIdx，可以认为是块内的线程偏移量（线程在线程块内的位置索引，有x,y,z三个分量，代表三个方向的偏移量，基于 uint3 定义，包含三个无符号整数的结构）
      • 核函数运行时，这两个变量是会变的，从而确定该线程的index
    • 坐标分量的范围
      • blockDim：块是怎么被分为线程的，最多可以定义成三维，否则其余维度补为1，dim3类型
      • gridDim：网格是怎样划分为块的，一般是二维，但是最多可以定义三维，dim3类型
      • blockDim/gridDim控制着划分逻辑，核函数在运行时，他们是不变的

  • dim3和uint3结构体

    • dim3是手工定义的，主机端可见，设备端也能查到。

    • uint3是设备端在执行的时候可见的，不可以在核函数运行时修改，初始化完成后uint3值就不变了。

    • 他们是有区别的。

软硬件映射

  • SM,SP是硬件结构；
    • 一个SM可以包含多个SP
  • GRID,BLOCK,THREAD是软件概念
    • thread是一个线程，多个thread组成一个线程块block，多个block又组成一个线程网格grid，另外，固定大小的threads可以被组织称一个warp,warp是实际执行时的最小单位，warp的指令相同，处理的数据不同。
    • block之间不能有数据依赖，可以利用SM提供的shared memory和__syncthreads()功能实现线程同步和通信。
  • 【重要】执行关系
    • 一个SM上，可以resident多个blocks（具体几个根据架构不同而不同，请参阅programming guide的附录），但是一个block必须resident到一个SM；
    • 如果线程占用的资源（core, 寄存器，内存等）少，那么block占用的资源就少，一个SM就可以resident多个blocks，提高并发度
    • 这些resident 在同一个SM上的blocks共享同该SM的寄存器，shared memory等资源。
    • SM会从当前的resident blocks里面以warp为单位抓取线程来进行计算。
    • 当前需要计算的指令，可以被SM的scheduler 发射到SP阵列上，也可以发射到SFU，以及访存。
    • 如果当前的warp由于访存等原因需要等待，将被切换出去，换一个就绪的warp进来继续计算，这样通过线程切换，用计算掩盖了访存等延迟

核函数

概述

  • 核函数就是在CUDA模型上诸多线程中运行的那段串行代码
  • 这段代码在设备上运行，用NVCC编译，产生的机器码是GPU的机器码，所以我们写CUDA程序就是写核函数

定义核函数

  • __global__ 关键字
限定符 执行 调用 备注
__global__ 设备端执行 主机或计算能力3以上设备调用 void返回类型
__device__ 设备端执行 设备端调用
__host__ 主机端执行 主机调用 限定符可以省略
  • 特殊情况
    • 特殊的情况就是有些函数可以同时定义为 device 和 host ，这种函数可以同时被设备和主机端的代码调用，那么就要告诉编译器，生成两份不同设备的机器码。

启动核函数

  • 把我们的CPU当做一个控制者，运行核函数
  • 这个三个尖括号’<<<grid,block>>>’内是对设备代码执行的线程结构的配置（或者简称为对内核进行配置），也就是线程结构中的网格，块
  • 实例
  • 核函数是同时复制到多个线程执行的
  • 当主机启动了核函数，控制权马上回到主机，而不是主机等待设备完成核函数的运行,也就是说是异步执行

同步

  • 显式主机等待设备端执行
    • cudaError_t cudaDeviceSynchronize(void);
  • 隐式，设备端不执行完，主机没办法进行
    • cudaError_t cudaMemcpy(void* dst,const void * src,
      size_t count,cudaMemcpyKind kind);
    • 当核函数启动后的下一条指令就是从设备复制数据回主机端，那么主机端必须要等待设备端计算完成

核函数限制

  • 只能访问设备内存
  • 必须有void返回类型
  • 不支持可变数量的参数
  • 不支持静态变量
  • 显式异步行为

CUDA执行模型

GPU架构

  • SM 流式多处理器

    • CUDA核心 core
    • 共享内存 share memory
    • L1 cache
    • 寄存器堆: Register File
    • 加载存储单元: LD/ST
    • 特殊功能单元: SFU

  • 一个 GPU上有多个SM，一个SM支持数百个线程并发执行

    • 当一个核函数的网格被启动的时候，多个block会被同时分配给可用的SM上执行。
    • 当一个blcok被分配给一个SM后，他就只能在这个SM上执行了，不可能重新分配到其他SM上
    • 多个线程块可以被分配到同一个SM上

  • 思考：一个SM不能容纳一个block的所有线程怎么办？

  • 并行度

    • SM上同一个块内的多个线程进行线程级别并行
    • 同一线程内，指令利用指令级并行将单个线程处理成流水线。

线程束

  • CUDA 采用单指令多线程SIMT架构管理执行线程，不同设备有不同的线程束大小，但是到目前为止基本所有设备都是维持在32。
  • 也就是说每个SM上有多个block，一个block有多个线程（可以是几百个，但不会超过某个最大值），但是从机器的角度，在某时刻T，SM上只执行一个线程束，也就是32个线程在同时同步执行
  • 线程束中的每个线程执行同一条指令，包括有分支的部分
  • 同一个SM上可以有不止一个常驻的线程束，有些在执行，有些在等待，他们之间状态的转换是不需要开销的。

软件与硬件逻辑映射

  • 并行度
    • 编程模型层面看所有线程都是并行执行的。
    • 在微观上看，所有线程块也是分批次的在物理层面的机器上执行，线程块里不同的线程可能进度都不一样，但是同一个线程束内的线程拥有相同的进度。
  • 同步
    • 并行可能会引起竞争，多线程以未定义的顺序访问同一个数据，就导致了不可预测的行为。
    • CUDA只提供了一种块内同步的方式，块之间没办法同步。
    • 同步是需要overhead的，所以块内并行所带来的时间减少一定要cover掉overhead

Fermi架构

  • 整体架构

    • 16个SM
    • 每个SM有32个core，每16个CUDA核心为一个组。有调度线程束的调度器和调度单元，还有共享内存，还有寄存器堆和一级缓存。
    • 每个核心都有一个全流水线的整数算数逻辑单元ALU，和一个浮点数运算单元FPU
    • 6个384-bits的GDDR5的DRAM内存接口
    • 支持6G的全局机载内存
    • 768KB的二级缓存，被所有SM共享
    • GigaThread，分配线程块到SM线程束调度器上

  • 加载/存储单元

    • 每个多处理器SM有16个加载/存储单元
    • 每个时钟周期内有16个线程（半个线程束）计算源地址和目的地址

  • 特殊功能单元SFU

    • 执行固有指令，如正弦，余弦，平方根和插值
    • SFU在每个时钟周期内的每个线程上执行一个固有指令。

  • 线程调度器

    • 每个SM有两个线程束调度器，和两个指令调度单元

    • 当一个线程块被指定给一个SM时，线程块内的所有线程被分成线程束

    • 两个线程束调度器选择其中两个线程束，再用两个指令调度器存储两个线程束要执行的指令

    • 当某个线程块被分配到一个SM上的时候，会被分成多个线程束，线程束在SM上交替执行

    • 每个线程束在同一时间执行同一指令，同一个块内的线程束互相切换是没有时间消耗的。

    • 思考：为什么一个SM内只有32核，每个warp32个线程，却要有两个warp scheduler？并且加载/存储单元只有16个而不是32个？

  • 并发执行kernel

    • 并发执行内核允许执行一些小的内核程序来充分利用GPU

Kepler架构

  • 整体架构
    • 强化的SM 即SMX，单个SM上CUDA核的数量，SFU的数量，LD/ST的数量等都提升了。
    • 动态并行
    • Hyper-Q技术

  • SMX

  • 内核可以启动内核
    • 可以完成递归操作，而不需借助CPU反复divide任务
  • Hyper-Q技术主要是CPU和GPU之间的同步硬件连接，以确保CPU在GPU执行的同事做更多的工作。Fermi架构下CPU控制GPU只有一个队列，Kepler架构下可以通过Hyper-Q技术实现多个队列

线程束

线程束和线程块

  • 线程束的划分
    • 线程束是SM中基本的执行单元，当一个网格被启动，等价于一个内核被启动
    • 每个内核对应于自己的网格，网格中包含线程块，线程块被分配到某一个SM上以后，将分为多个线程束
    • 每个线程束一般是32个线程（目前的GPU都是32个线程，但不保证未来还是32个）
    • 在一个线程束中，所有线程按照单指令多线程SIMT的方式执行，每一步执行相同的指令，但是处理的数据为私有的数据

线性地址计算

  • 层次关系

    • 一个kernel对应一个网格
    • 一个网格对应3维block (blockIdx.x,blockIdx.y,blockIdx.z)
    • 一个block对应3维thread (threadIdx.x,threadIdx.y,threadIdx.z)

  • 轴优先级

    • 普通数组轴优先级是从右边到左
    • cuda轴优先级是从左到右
    • 例如，某个线程(threadIdx.x,threadIdx.y,threadIdx.z)对应的数组元素是t[z][y][x];

  • 线性地址计算
    $$tid=threadIdx.x+threadIdx.yblockDim.x+threadIdx.zblockDim.x*blockDim.y$$

  • 一个线程块包含的线程束的个数

    • ceil(threadsPerBlock / warpSize)

线程束分化

  • 指令广播

    • 每个SM上执行的基本单位是线程束，也就是说，单指令通过指令调度器广播给某线程束的全部线程，这些线程同一时刻执行同一命令

  • 线程束被执行的时候会被分配给相同的指令，处理各自私有的数据

  • 如果一个线程束中的不同线程包含不同的控制条件，那么当我们执行到这个控制条件是就会面临不同的选择

  • CPU中右分支预测技术，比GPU要高级得多。CPU适合逻辑复杂计算量不大的程序，比如操作系统，控制系统，GPU适合大量计算简单逻辑的任务

  • 概念：当一个线程束的32个线程执行这段代码的时候，如果其中n个执行if中的代码段，而另外32-n个执行else中的代码块，那么就说这个线程束发生了分化。

  • 【思考】：不是说同一个线程束只能执行同一种指令吗，那怎么解决线程束分化问题呢？

    • 每个线程都执行所有的if和else部分
    • 当第一个条件成立的时候，执行if块内的代码，其余线程啥也不干
    • 当另一个条件成立的时候，执行else块内的代码，其余线程啥也不干
    • 这两个条件都执行完了，再进行下一轮也就是下一个指令

线程束分化解决方法

  • 根本思路是避免同一个线程束内的线程分化，而让我们能控制线程束内线程行为的原因是，线程块中线程分配到线程束是有规律的而不是随机的

  • 当一个线程束中所有的线程都执行if或者，都执行else时，不存在性能下降；

  • 线程束内的线程是可以被我们控制的，那么我们就把都执行if的线程塞到一个线程束中，或者让一个线程束中的线程都执行if，另外线程都执行else

  • 注意：不要看kernel函数里有if语句就一定会发生分化，主要是看每个线程块被分成的线程束，每个线程束到底执行了几种指令分支

资源分配

  • 一个SM上被分配多少个线程块和线程束取决于SM中可用的寄存器和共享内存，以及内核需要的寄存器和共享内存大小。
  • 寄存器资源分配

  • 共享内存分配

  • 当kernel占用的资源较少，那么更多的线程（这是线程越多线程束也就越多）处于活跃状态，相反则线程越少处于活跃状态

  • 当SM内的资源没办法处理一个完整块，那么程序将无法启动

活跃线程数

  • 当寄存器和共享内存分配给了线程块，这个线程块处于活跃状态，所包含的线程束称为活跃线程束。
    • 选定的线程束
      • 当SM要执行某个线程束的时候，执行的这个线程束叫做选定的线程束
    • 符合条件的线程束
      • 准备要执行的叫符合条件的线程束，必须满足：
      • 32个CUDA核心可以用于执行
      • 执行所需要的资源全部就位
    • 阻塞的线程束
      • 如果线程束不符合条件还没准备好就是阻塞的线程束。

延迟隐藏

  • 延迟是从任务开始到任务结束的时间。延迟隐藏实际上是通过一段时间内执行了多个任务，使得一个任务的平均执行时间变短了，好像是隐藏了计算延迟。
  • 即最大化功能单元的利用率。利用率与常驻线程束直接相关。
  • 延迟
    • 算术指令延迟：算术操作从开始，到产生结果之间的时间。这个时间段内只有某些计算单元处于工作状态，而其他逻辑计算单元处于空闲。大概10~20 个时钟周期。
    • 内存指令延迟：当产生内存访问的时候，计算单元要等数据从内存拿到寄存器，这个周期是非常长的，大概400~800 个时钟周期。
  • 隐藏延迟
    • 比如32个单精度浮点计算器，每次计算延迟20个时钟周期，那么我需要最少 32x20 =640 个线程使设备处于忙碌状态。在前20个时钟周期内，每个时钟周期都能把所有的硬件资源占满，如果后续持续不断有线程再加入，那么所有硬件依然是持续忙碌没有空闲的状态

占用率

  • 占用率是一个SM种活跃的线程束的数量，占SM最大支持线程束数量的比

同步

    • __syncthread();
    • 这个函数只能同步同一个块内的线程，不能同步不同块内的线程，

CUDA内存模型

GM107中SMM的内部结构

  • 指令缓存（Instruction Cache）存放核函数中的的指令。
  • 指令缓冲区（Instruction Buffer）Core访问它的速度比指令缓存更快，但是空间更小。
  • 线程束调度器（Warp Scheduler）调度各个线程束执行、挂起等。
  • 流处理器（Core 或 Stream Processor）
  • 一级缓存（L1 Cache）、纹理内存（Texture），他们公用同一片cache区域，可以通过调用CUDA函数设置各自的所占比例。
  • 共享内存（Shared Memory）
  • 寄存器区（Register File）供各条线程在执行时存放临时变量的区域。

寄存器

  • 寄存器无论是在CPU还是在GPU都是速度最快的内存空间
  • 寄存器对于每个线程是私有的，寄存器通常保存被频繁使用的私有变量，
  • 寄存器是SM中的稀缺资源，Fermi架构中每个线程最多63个寄存器。Kepler结构扩展到255个寄存器
  • 一个线程如果使用更少的寄存器，那么就会有更多的常驻线程块，SM上并发的线程块越多，效率越高，性能和使用率也就越高
  • 如果一个线程里面的变量太多，以至于寄存器完全不够呢？这时候寄存器发生溢出，本地内存就会过来帮忙存储多出来的变量，这种情况会对效率产生非常负面的影响

本地内存

  • Local Memory也是每个线程私有的
  • 本地内存一般位于片内存储体中，在核函数编写不恰当的情况下会部分位于片外存储器中
  • 在核函数中符合存储在寄存器中但不能进入核函数分配的寄存器空间中的变量将被存储在Local Memory中，Local Memory中可能存放的变量有以下几种：
    • 使用未知索引的本地数组
    • 较大的本地数组或结构体
    • 任何不满足核函数寄存器限定条件的变量

共享内存（Shared memory）

  • __share__关键字修饰的内存
  • 位于每个SM中，其访问速度仅次于寄存器
  • 每个SM都有一定数量的由线程块分配的共享内存，特点是一个线程块中的所有线程都可以访问。主要存放频繁修改的变量。
  • 过度使用共享内存，而导致SM上活跃的线程束减少，也就是说，一个线程块使用的共享内存过多，导致很多线程块没办法被SM启动，这样影响活跃的线程束数量。
  • SM中的一级缓存，和共享内存共享一个64k的片上内存，可以手动设置
  • bank划分
    • 比较新的GPU中，一个Warp即32个SP可以同时访问32个Bank
    • 如果是读操作，那么GPU底层可以通过广播的方式将数据传给各个Thread，延迟不会很大
    • 如果是写操作，就必须要等上一个线程写完成后才能进行下一个线程的写操作，延时会比较大

常量内存(Constant Memroy)

  • 常量内存位于每个流处理器（SM）中和片外的RAM存储器中

    • 当常量内存被初始化的时候，它会先全部保存在片外的RAM中，然后使用8KB（具体数值随显卡型号变化）的一级缓存 作为缓冲，可以将访问延迟（Latency）缩到很短

  • 主机端代码是可以初始化常量内存的，被主机端初始化后不能被核函数修改，但是对同一个变一单元中的所有核函数可见

  • 读取机制

    • 一次读取会广播给所有线程束内的线程。

  • 当线程束中所有线程都从相同的地址取数据时，常量内存表现较好

    • 比如执行某一个多项式计算，系数都存在常量内存里效率会非常高，
    • 但是如果不同的线程取不同地址的数据，常量内存就不那么好

纹理内存(Texture Memory)

  • 位于每个流处理器（SM）中和片外的RAM存储器中
  • 纹理内存，顾名思义就是专门用于纹理贴图操作的，故在该操作上使用纹理内存更加高效。
    • 在进行纹理贴图操作或几何表面访问的操作时，由于访问数据的方式不符合访问全局内存和常量内存的访问规律，导致访问速度下降，但是纹理内存可以解决这一问题
  • 纹理内存具有广播机制，和常量内存很相似。

全局内存（Global memory）

  • 位于片外存储体中。容量大、访问延迟高、传输速度较慢。在2.X计算力之后的GPU上，都会使用二级缓存（L2 cache）做缓冲，达到较快的传输速度，但这并不能减少访问的延迟（Latency）
  • 全局内存，就是我们常说的显存，就是GDDR的空间，全局内存中的变量，只要不销毁，生命周期和应用程序是一样的。 全局内存对应于设备内存，一个是逻辑表示，一个是硬件表示、
  • GPU上最大的内存空间，延迟最高。
  • 全局内存可以动态声明，或者静态声明，可以用下面的修饰符在设备代码中静态的声明一个变量
    • __device__

缓存

  • 每个SM都有一个一级缓存，所有SM公用一个二级缓存，GPU读操作是可以使用缓存的，但写操作不能被缓存。
  • “缓存一致性”（cache coherency）
    • 缓存一致是指一个内存的写操作需要通知所有核的各个级别的缓存，因此，无论何时，所有处理器核看到的内存视图是完全一样的。随着处理器中核数量的增多，这个“通知”的开销迅速增大，使得“缓存一致性”成为限制一个处理器中核数不能太多的一个重要因素。
    • CPU遵循“缓存一致”原则，而GPU不是。在GPU中系统不会自动的更新其他核的缓存。
